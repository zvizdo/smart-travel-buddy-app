# Implementation Plan: DAG Timeline View

**Branch**: `002-dag-timeline-view` | **Date**: 2026-04-01 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-dag-timeline-view/spec.md`

## Summary

Build a vertical timeline view as an alternative to the existing map view for trip itineraries. The timeline shows the same DAG (nodes and edges) organized chronologically — top is earliest, bottom is latest — with node blocks sized proportionally to duration (like Google Calendar). The implementation is entirely frontend: a pure layout algorithm computes positions from existing Firestore data, 7 new React components render the timeline, and 2 new state variables in `page.tsx` integrate it with the existing view/interaction architecture. No backend changes are required.

**Scale/Scope**: Typical trips: 20-100 nodes, up to 4 parallel lanes, 10-day spans

## Architecture Overview

### Data Flow

```
Firestore (nodes, edges)
    │ onSnapshot (real-time)
    ▼
useTripNodes / useTripEdges (existing hooks)
    │
    ▼
page.tsx (existing state: selectedNodeId, selectedEdgeId, pathMode, canEdit)
    │ + NEW: viewMode, timelineZoom
    │
    ├─── TripMap (hidden when timeline active)
    │
    └─── TimelineView
           │
           ├── useMemo: computeTimelineLayout(nodes, edges, pathResult, zoom)
           │     → TimelineLayout { lanes[], dateMarkers[], totalHeightPx }
           │
           ├── TimelineDateGutter (sticky left column, 56px)
           │
           └── TimelineLane[] (right column, flex-1)
                 ├── TimelineNodeBlock[] (tap → onNodeSelect → NodeDetailSheet)
                 ├── TimelineEdgeConnector[] (tap → onEdgeSelect → EdgeDetail)
                 └── GapRegion indicators (compressed idle periods)
```

### State Integration

Only 2 new state variables added to `page.tsx`:


All existing state and callbacks are shared between views:
- `selectedNodeId` / `selectedEdgeId` — selection synced across views
- `pathMode` / `setPathMode` — path filter affects lane rendering
- `handleNodeSelect` / `handleEdgeSelect` — opens same detail sheets
- `canEdit` — same permission model
- `addNodePlace` / `insertEdgeId` — same AddNodeSheet mechanics

### View Mounting Strategy

Both `TripMap` and `TimelineView` remain mounted. The inactive view is hidden via `display: none` (Tailwind `hidden` class). This preserves:
- Google Maps SDK state (tile cache, camera position, projection)
- Timeline scroll position
- No re-initialization cost on view switch

### Layout Algorithm (`computeTimelineLayout`)

9-step pure function:

1. **Determine active lanes** — single lane in "all" mode, per-participant lanes in "mine" mode
2. **Resolve missing times** — interpolate from DAG neighbors, mark `hasMissingTime`
3. **Compute px/ms scale** — from zoom level (`[8, 16, 32, 60, 120]` px/hour)
4. **Compute Y-offsets** — `(arrival_ms - earliest_ms) / 3_600_000 * px_per_hour`
5. **Compute block heights** — `max(56, duration_hours * px_per_hour)`
6. **Detect and compress gaps** — replace >8h idle periods with 40px indicator
7. **Compute connector heights** — `max(40, travel_hours * px_per_hour)`
8. **Detect timing warnings** — `deficit > 10 min` = warning
9. **Compute date markers** — emit at day boundaries per node timezone
